# Raise Lifecycle

Every Backed raise moves through a predictable sequence.

## Lifecycle stages

| Stage | Description |
| --- | --- |
| Identity | An ERC8004 agent identity already exists |
| Configuration | The project’s economic and metadata assumptions are prepared |
| Creation | The factory creates the project and related addresses |
| Approval | An authorized operator approves the project |
| Commitment | Participants commit USDM during the sale window |
| Resolution | The sale finalizes or fails |
| Post-close | Claims, refunds, and redemptions become available according to state |

## Stage-by-stage interpretation

### Identity

Before anything else, the relevant agent identity already exists. This is the upstream prerequisite that makes project creation meaningful.

### Configuration

At this stage, the economic and messaging assumptions of the raise are defined. If the configuration is still unstable, the raise is not yet ready for creation.

### Creation

Creation establishes the project onchain. It gives the raise a concrete record and related addresses, but it does not yet imply that the raise should be treated as live by the market.

### Approval

Approval changes the meaning of the project. The raise is no longer merely created; it is now approved for launch according to the operator model of the platform.

### Commitment

This is the stage at which the raise becomes economically active for participants. USDM commitments matter here, and the live sale window becomes central.

### Resolution and post-close

After the sale window, the relevant questions shift. The focus is no longer “can capital be committed?” but “what does the resolved state imply for claims, refunds, and later ownership behavior?”

## The distinction that matters most

Backed separates:

- project existence,
- project launch,
- project funding,
- project settlement.

That separation is intentional. It lowers ambiguity and makes responsibility explicit.

## Operating rule

Do not use “launch” as a catch-all term. In Backed, creation and approval are different events with different implications for founders, investors, and operators.
