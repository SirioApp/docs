# Founder Guide

This guide is for the team initiating a raise.

It is written to help founders think in terms of launch discipline rather than simple form completion. A founder using Backed is not just filling in metadata. They are preparing a project that will later be interpreted by investors, approved by operators, and settled by contracts.

## What a founder is actually doing

At a practical level, a founder is not just publishing a listing. A founder is creating an onchain project that will later be approved, funded, and settled according to contract state.

## Launch readiness standard

Before creating a raise, a founder should be able to answer all of the following clearly:

- Which agent identity is this raise attached to?
- Which environment is the raise targeting?
- Which asset will participants commit?
- What timing assumptions define the sale window?
- What message should the market see only after approval is complete?

If those questions are not already clear before creation, the project is not yet launch-ready.

## Founder checklist

<Steps>
  <Step title="Confirm the agent identity">
    Make sure the intended ERC8004 agent ID already exists and resolves to the expected wallet.
  </Step>
  <Step title="Prepare the raise configuration">
    Finalize project name, description, categories, collateral assumptions, launch timing, duration, and token metadata before the project is created.
  </Step>
  <Step title="Create the project">
    Use the application or controlled script flow to create the raise and capture the resulting project ID.
  </Step>
  <Step title="Wait for approval">
    Do not treat the project as launched, or communicate it as live, until an authorized operator approves it.
  </Step>
  <Step title="Monitor the live raise">
    During the sale window, monitor commitments, timing, and the observable sale state.
  </Step>
</Steps>

## What good founder behavior looks like

Good founder behavior in Backed is mostly about timing and clarity.

The project should be created only after the important inputs are stable.

The launch message should be coordinated with approval, not merely with creation.

The live period should be monitored with the understanding that investors are reacting to the onchain sale state, not just to the narrative around the project.

## Common founder mistakes

| Mistake | Why it matters |
| --- | --- |
| Using the wrong agent ID | The raise may point at the wrong identity or fail entirely |
| Finalizing metadata too late | Operational confusion increases after creation |
| Promoting before approval | The market may see a project that is not yet live |
| Confusing testnet with mainnet | All addresses and state interpretations become unreliable |

## Coordination with operators

Founders and operators should align on at least four things before launch communication begins:

- the intended environment,
- the project identifier,
- the expected approval timing,
- the public-facing link or reference users will see.

<Tip>
  Treat approval as part of launch planning rather than as a minor post-creation formality.
</Tip>
