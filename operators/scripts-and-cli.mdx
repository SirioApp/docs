# Scripts and CLI

Backed exposes two operator-facing surfaces:

- repository scripts used from this frontend workspace;
- the `backed` CLI used for inspection and explicit protocol operations.

## Environment variables

### CLI environment variables

| Variable | Purpose |
| --- | --- |
| `BACKED_NETWORK` | Selects `testnet` or `mainnet` |
| `BACKED_RPC_URL` | Overrides the default RPC URL |
| `BACKED_FACTORY` | Overrides the resolved factory address |
| `BACKED_ALLOWLIST` | Overrides the resolved allowlist address |
| `BACKED_PRIVATE_KEY` | Preferred secret for write operations |
| `PRIVATE_KEY` | Fallback secret when the CLI-specific key is unset |

### Repository script environment variables

| Variable | Purpose |
| --- | --- |
| `PRIVATE_KEY` | Required signer for write operations |
| `AGENT_ID` | Required when creating a raise |
| `PROJECT_ID` | Required when approving an existing raise |
| `NEXT_PUBLIC_MEGAETH_RPC_URL` | Optional RPC override |
| `COLLATERAL_ADDRESS` | Optional collateral override, defaults to USDM |

## Repository scripts

| Script | Purpose |
| --- | --- |
| `npm run create-raise` | Creates a raise using environment-driven configuration |
| `npm run approve-raise` | Approves a project so it becomes available as a live raise |
| `npm run check-raise` | Inspects an existing raise configuration and status |

## Representative CLI commands

### Network

```bash
backed --network testnet network
```

### Factory

```bash
backed factory info
backed factory project <id>
backed factory create --agent-id 0 --name "..." --token-name T --token-symbol T
backed factory approve <id>
```

### Sale

```bash
backed sale status --project-id <id>
backed sale approve-collateral --project-id <id> <amount>
backed sale commit --project-id <id> <amount>
backed sale finalize --project-id <id>
```

### Allowlist

```bash
backed allowlist info
backed allowlist is-allowed <address>
backed allowlist add <address>
```

## Operator standard

Use the product UI for discovery and ordinary user flows. Use scripts or the CLI when:

- the action is privileged;
- the action must be repeated or scripted;
- the environment or signer must be fully explicit;
- the result needs direct operational verification.
