# Security Model

Backed is contract-first, but not trustless in the simplistic sense.

Its safety depends on both:

- the integrity of the deployed contracts,
- the discipline of operator workflows.

This distinction matters because the platform does not claim that every important decision is automatically safe just because it is exposed onchain. Certain decisions remain operational by nature, and the documentation should make those boundaries visible.

That honesty matters even more because Backed is exploring the AAO category rather than presenting it as a fully solved problem. The platform should be trusted to the extent that its state boundaries, operational controls, and disclosure standards are credible, not because the category is already complete.

## Core security boundaries

| Boundary | What it protects |
| --- | --- |
| Onchain state boundary | Approval, commitments, settlement, claims, refunds |
| Operator boundary | Privileged actions such as approval and explicit writes |
| Treasury boundary | Restriction of treasury-controlled execution paths |
| Environment boundary | Separation of testnet and mainnet targeting |

## Practical trust assumptions

Users should assume:

- the contracts define the current truth,
- the manifests correctly resolve the intended contracts,
- operators are using the right signer on the right environment,
- collateral assumptions are correctly configured.

## What this means in practice

Backed reduces ambiguity by moving critical lifecycle decisions onchain, but it does not remove the need for disciplined operations. Approval, environment selection, and privileged writes remain meaningful trust surfaces.

In short: Backed should be read as a serious operating system for an emerging category, not as proof that the category is already frictionless or fully autonomous.
