# How Backed Works

Backed is easiest to understand as a layered operating model.

| Layer | Role |
| --- | --- |
| Identity | Resolves the agent referenced by the raise |
| Factory | Creates projects, stores project metadata, and gates approval |
| Sale | Manages commitments, finalization, claims, and refunds |
| Treasury controls | Restrict sensitive execution paths |
| Interfaces | Application, scripts, and CLI surfaces used by different actors |

## Two ways to read the platform

There are two useful ways to understand Backed.

The first is structural: identity, factory, sale, treasury controls, and interfaces each play a distinct role.

The second is chronological: a raise moves from identity prerequisite to configuration, creation, approval, commitment, resolution, and post-close ownership outcomes.

Most confusion comes from mixing these two views. A reader sees a project in the interface and assumes it is already fully launched, when what they are really seeing may only be one stage in a broader lifecycle.

## Core flow

<Steps>
  <Step title="Identity already exists">
    A valid agent identity must already exist in the ERC8004 Identity Registry before the raise can be opened.
  </Step>
  <Step title="A project is configured">
    The founder or operator defines project metadata, collateral assumptions, timing, and token configuration.
  </Step>
  <Step title="The project is created">
    The factory creates the project onchain and establishes the addresses needed for the raise lifecycle.
  </Step>
  <Step title="The project is approved">
    An authorized operator approves the project before it should be treated as live or marketed as launched.
  </Step>
  <Step title="Capital is committed">
    Investors approve USDM and commit capital during the active sale window.
  </Step>
  <Step title="The sale resolves">
    Finalization, refundability, claims, and later redemption behavior are determined by sale and vault state.
  </Step>
</Steps>

## What changes at each stage

When a raise moves from one stage to another, the meaning of the project changes.

- Before creation, there is no project-level raise state.
- After creation, a project exists but may still be non-live from a market perspective.
- After approval, the project can be treated as launched.
- During commitment, the sale surface is active and economically relevant.
- After resolution, participants should reason about claims, refunds, and ownership according to the contract-defined outcome.

## Why the model is structured this way

<AccordionGroup>
  <Accordion title="Identity is a prerequisite">
    Backed does not treat identity creation as an implicit side effect of opening a raise. Identity is upstream and explicit.
  </Accordion>
  <Accordion title="Creation is not launch">
    A project can exist onchain before it is approved. Approval exists to preserve a deliberate launch boundary.
  </Accordion>
  <Accordion title="Interfaces do not own state">
    Interfaces can expose state and facilitate action, but they do not define lifecycle truth. The contracts do.
  </Accordion>
  <Accordion title="Environment is a real risk boundary">
    Testnet and mainnet are separate deployment surfaces. Wrong-environment reads and writes are not cosmetic mistakes.
  </Accordion>
</AccordionGroup>

## Practical implication

Anyone evaluating a raise should separate four different questions:

- Does the project exist?
- Has it been approved?
- Is the sale currently active?
- What does the current sale state imply for capital, claims, or refunds?

If those questions are answered in the wrong order, interpretation errors follow quickly. A project that exists may be treated as if it were already launched. A raise that is visible may be mistaken for one that is actively open. A close event may be mistaken for finalized settlement. The purpose of Backed’s structure is to reduce those ambiguities.
