# How Backed Works

Backed is easiest to understand as a layered system.

| Layer | Role |
| --- | --- |
| Identity | Resolves the agent referenced by a raise |
| Factory | Creates projects, stores metadata, and gates approval |
| Sale | Handles commitments, finalization, claims, and refunds |
| Treasury controls | Restrict operational execution paths |
| Interfaces | UI, scripts, and CLI used by different actors |

## Core flow

<Steps>
  <Step title="Identity already exists">
    A valid agent identity must already exist in the ERC8004 Identity Registry.
  </Step>
  <Step title="Project is configured and created">
    The factory creates a project using agent, metadata, collateral, timing, and token settings.
  </Step>
  <Step title="Project is approved">
    An authorized operator explicitly approves the project before it should be treated as live.
  </Step>
  <Step title="Investors commit capital">
    Participants approve USDM and commit into the sale during the active window.
  </Step>
  <Step title="The raise settles">
    Claims, refunds, and later redemption behavior are determined by the sale and vault state.
  </Step>
</Steps>

## Design principles

<AccordionGroup>
  <Accordion title="Identity is a prerequisite, not a side effect">
    Backed does not pretend that creating a raise also creates a valid identity. Identity is upstream and explicit.
  </Accordion>
  <Accordion title="Creation is not launch">
    A project can exist onchain before it is approved. Approval exists to enforce a deliberate launch boundary.
  </Accordion>
  <Accordion title="Interfaces do not own state">
    The application, scripts, and CLI read and write against the contracts. They do not define lifecycle truth.
  </Accordion>
  <Accordion title="Environment selection is operationally significant">
    Testnet and mainnet are separate deployment surfaces. Wrong-environment reads and writes are meaningful failures.
  </Accordion>
</AccordionGroup>
