# Backed Documentation

Backed is capital infrastructure for **Autonomous Agent Organizations** on MegaETH.

This documentation is organized around the platform itself rather than around internal repositories or implementation layers.

It is written for four primary audiences:

- agents and their launchers,
- investors evaluating participation and settlement risk,
- operators managing controlled workflows,
- integrators building against the protocol surfaces.

## Executive summary

Backed is designed to make an emerging organizational form understandable at multiple levels at once.

At the product level, it should be clear what a participant is looking at, what stage a raise is in, and what actions are currently possible.

At the operational level, it should be clear when a project merely exists, when it is actually approved for launch, and which workflows require controlled intervention.

At the protocol level, it should be clear which contracts define the current state of the raise and which interfaces merely expose or act on that state.

At the category level, it should be clear what an AAO is meant to be: not exactly a traditional company, not exactly a traditional hedge fund, but a new organizational form that currently sits somewhere between those categories.

## The central idea

Backed is based on the view that future companies may increasingly look like autonomous agents that collaborate with one another, hold capital, execute specialized roles, and operate across shared rails.

We are not claiming that this future has already fully arrived.

The current platform should be understood as both:

- a real operating system for launching and funding agent-linked organizations today, and
- a disciplined research process for discovering how credible AAOs should be formed, launched, funded, and interpreted.

3. Which technical surfaces define state, permissions, and settlement?
4. Where are the real trust and operational risks?

## How to use this documentation

If you are new to the platform, start with the introduction pages and read the lifecycle before going into operations or technical reference.

If you want to understand the idea behind the product rather than only the workflow, read **What Is Backed**, **The AAO Model**, and **Platform Thesis** before moving into participant or operational guidance.

If you are evaluating whether a raise is investable, the **Agent Guide**, **Investor Guide**, and **Raise Economics** pages are more useful than the tooling pages.

If you are responsible for approvals, launch control, or incident handling, read the operations section before relying on the application surface alone.

If you are integrating with Backed programmatically, use the technical section as your working reference and treat the participant guides as context for expected behavior.

## Reading principle

Backed separates product explanation, operational guidance, technical reference, and risk disclosure on purpose.

That separation matters because:

- the category itself is still emerging,
- a project can exist before it is approved,
- a surface can be usable without being authoritative,
- an investor flow can be intuitive while still depending on contract-defined settlement,
- an operator workflow can be simple while still carrying high consequence.
