# Platform Overview

Backed enables capital raises for autonomous organizations whose treasury and lifecycle are enforced onchain.

At a platform level, Backed combines four concerns:

- identity, so a raise is attached to an existing agent;
- issuance, so a project can be created and approved;
- capital formation, so investors can commit collateral;
- settlement, so claims, refunds, and post-raise ownership resolve from contract state.

## The platform in one sentence

Backed turns an approved agent identity into an investable onchain raise with controlled treasury execution and deterministic settlement paths.

## System model

The system is easiest to understand as five connected layers:

| Layer | Role |
| --- | --- |
| Identity | Confirms the agent identity referenced during raise creation |
| Factory | Creates projects, stores project metadata, and gates approval |
| Sale | Manages commitments, close conditions, claims, and refunds |
| Treasury controls | Restrict treasury execution through Safe wiring and allowlisting |
| Interfaces | Product UI, scripts, and CLI used by founders, investors, and operators |

<Note>
  Backed is not organized around a single application surface. The product UI, scripts, and CLI are all execution surfaces over the same onchain system.
</Note>

## Why approval is a separate step

Backed deliberately separates project creation from project approval.

That distinction matters for governance and safety:

- creation proves that a project exists;
- approval determines whether it should be treated as launched;
- interfaces must not present creation alone as sufficient for public availability.

## What Backed is not

Backed is not a generic crowdfunding frontend.

It is also not a trustless abstraction that removes all operator responsibility. Operators still choose environments, hold privileged keys, approve projects, and manage treasury boundaries. The platform reduces ambiguity, but it does not eliminate operational risk.
